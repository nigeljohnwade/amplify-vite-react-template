## Commands

- `npm run dev` — start the Vite dev server
- `npm run build` — type-check (`tsc`) then build for production (`vite build`)
- `npm run lint` — ESLint over `ts`/`tsx` (`--max-warnings 0`, so any warning fails)
- `npm run preview` — preview the production build
- `npx ampx sandbox` — deploy a personal cloud sandbox backend and watch for changes; this generates `amplify_outputs.json` (see below)
- `npx ampx sandbox delete` — tear down the sandbox

There is no test runner configured in this project.

## Prerequisites for running locally

The app will not start without two pieces of local-only config (both git-ignored):

1. **`amplify_outputs.json`** at the repo root. Imported by both `src/main.tsx` and `src/App.tsx` to configure Amplify. It is generated by running `npx ampx sandbox` (or by an Amplify pipeline deploy). Without it, `npm run dev` fails to compile.
2. **`VITE_MAPBOX_API_KEY`** environment variable (e.g. in a `.env.local`). Read in `src/App.tsx` to initialize Mapbox GL. Without it the map renders blank.

## Architecture

This is an **AWS Amplify Gen 2** fullstack app: a single-page React 19 + Vite frontend backed by an Amplify backend defined in code under `amplify/`.

### Backend (`amplify/`)

- `backend.ts` wires together the `auth` and `data` resources via `defineBackend`.
- `auth/resource.ts` — Cognito user pool, email login only.
- `data/resource.ts` — the GraphQL/AppSync + DynamoDB schema. The data client's TypeScript types flow from here: `ClientSchema<typeof schema>` is exported as `Schema` and imported on the frontend as `import type { Schema } from '../amplify/data/resource'`. **Editing this file changes the frontend's types** — the two are coupled.
  - Models: `Plan` (the core entity), `Setting`, `Category`. `Location` is a `customType` (not a model) embedded in `Plan`.
  - Authorization is `userPool` only with `allow.owner()` — records are scoped to the signing-in user. API-key/public auth has been intentionally removed (see commented-out code).

### Frontend (`src/`)

- `main.tsx` — entry point. Wraps the app in `<Authenticator>` (Amplify hosted-UI login gate) and a `react-router` `<BrowserRouter>`. Defines the routes: `App` is the layout, with nested pages `/` (index → `PlanList`), `/create` (`CreatePlan`), and `/update/:id` (`UpdatePlan`).
- `client.ts` — calls `Amplify.configure(outputs)` and exports the single typed data client (`generateClient<Schema>()`). Configuration lives here (not in `main.tsx`/`App.tsx`) so the client is always created after Amplify is configured, regardless of import order. **Import the client from here**; do not call `generateClient` elsewhere.
- `App.tsx` — the **layout** (not a page). It owns the shared concerns and renders the active page via `<Outlet>`:
  - Subscribes to `client.models.Plan.observeQuery()` for live-updating data (mutations live in the page components via `client.models.Plan.create/update/delete`).
  - Manages a Mapbox GL map via refs. The map's center coordinate is the source of truth for a plan's saved location (the "Save location" checkbox stores the current map center into the `Location` field).
  - Shares `{ plans, center, flyTo }` with pages through Outlet context — see `context.ts` (`PlanContext` type + the `usePlanContext()` hook). `flyTo` is memoized so pages can safely depend on it in effects.
- `pages/` — one routed component per directory, each reads what it needs from `usePlanContext()`:
  - `PlanList` — list/tile view (local `tileView` toggle); Edit/New are `<Link>`s; `flyTo(INITIAL_CENTER)` on mount.
  - `CreatePlan` — renders `PlanForm`, calls `Plan.create`, then navigates to `/`.
  - `UpdatePlan` — fetches the plan by route param via `Plan.get({ id })` (so deep-links/refreshes work rather than relying on the in-memory list), flies to its location, calls `Plan.update`.
- `components/molecules/PlanForm/` — shared uncontrolled form used by both `CreatePlan` and `UpdatePlan`. Extracts fields via `FormData` and hands a normalized `PlanInput` to the page's `onSubmit`. The longitude/latitude inputs are display-only; the saved location always comes from the current map `center` plus the "Save location" checkbox.
- `components/atoms/` — small presentational primitives, each colocated with its CSS:
  - `Stack` — vertical layout with `spacing` of `content | components | containers`.
  - `InputGroup` — wraps a label + input.
  - `InteractionControl` — unified button/link. Renders an `<a>` when `href` is set, otherwise a `<button>` (`type` defaults to `button`). `variant` (`linkButton | textButton | iconButton`) selects the CSS class. Prefer this over raw `<button className="button">`. Note: in-app navigation still uses react-router `<Link>`, not this component's `href`.
- `constants.ts` — `INITIAL_CENTER`, `INITIAL_ZOOM`, and the hardcoded `categories` list (note: distinct from the unused `Category` data model).

### Styling

Plain CSS (no framework). `src/index.css` and `src/App.css` define CSS custom properties used across components; atom components import their own `.css` files. Plan list items get `wide`/`tall` modifier classes based on whether they have a location/date.

## Deployment

`amplify.yml` defines the Amplify hosting pipeline: backend deploys via `npx ampx pipeline-deploy`, frontend builds with `npm run build` and serves from `dist/`.
